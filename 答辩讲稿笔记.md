## 第 5 页 / 核心功能：RBAC 权限体系

> **PPT 内容**：登录流程 + 角色权限树截图

### 讲什么

这是项目的核心亮点之一——**菜单级 + 按钮级** 双重权限控制。

### 关键代码

```tsx
// router/auth.tsx — 路由守卫
const AuthGuard = ({ children }) => {
  const { token, permissionList } = state;

  if (!token || permissionList.length === 0) {
    return <Navigate to="/login" />;
  }
  return <>{children}</>;
};
```

**代码解释**：
- 这个守卫做了**两层校验**：第一层检查 token 是否存在（是否登录），第二层检查 permissionList 是否有内容（权限数据是否已加载）
- 只有两个条件同时满足，才放行到主页面
- 否则直接重定向到 `/login`，并且 `replace` 不留下历史记录

```tsx
// Sidebar.tsx — 菜单级权限
const buildMenuItems = () => {
  return topMenus
    .filter((m) => hasPermission(m.permission))  // 过滤无权限菜单
    .map((m) => ({
      key: m.path,
      icon: iconMap[m.icon],
      label: m.title,
    }));
};
```

**代码解释**：
- 菜单是**动态渲染**的，不是写死的左侧栏
- `hasPermission` 检查当前角色的权限列表里是否包含这个菜单的 `permission` 标识
- 没有权限的菜单直接 `filter` 掉，用户根本看不到入口

```tsx
// 按钮级权限 —— 后端返回类似这样的权限标识列表
permissionList: ['system:user', 'system:user:add', 'system:user:edit']

// 然后在前端这样用
{hasPermission('system:user:add') && (
  <Button>新增用户</Button>
)}
```

**代码解释**：
- 按钮权限更精细，不是隐藏菜单，而是控制页面内的某个按钮是否渲染
- 比如"新增用户"这个按钮，只有在 `permissionList` 包含 `system:user:add` 时才显示
- 超级管理员拥有 `['*']` 通配符，所有按钮可见

### 状态持久化

```tsx
// store/index.tsx — 登录后持久化到 localStorage
case 'SET_PERMISSION_LIST':
  if (action.payload?.length > 0) {
    storage.set('permissionList', action.payload);
  }
  return { ...state, permissionList: action.payload };
```

**为什么这样做**：
- 第一次实现时发现：刷新页面后 `permissionList` 恢复到 `[]`，但 `token` 还在，结果页面进了但菜单渲染不出来
- 解决方案：把 `permissionList` 也持久化到 `localStorage`，刷新后完整恢复
- 然后在 `AuthGuard` 里加了 `permissionList.length === 0` 的判断——即使 `token` 存在，权限数据没加载也强制回登录页

**口述稿**：
> "权限体系分两层。第一层是菜单级，不同角色登录后看到的左侧菜单不一样，运营经理看不到系统管理，客服看不到订单聚合。第二层是按钮级，同一个页面内，不同角色看到的操作按钮不同，比如只有超级管理员能看到'删除'按钮。
>
> 这里有一个踩坑的点：最开始权限列表没有做持久化，刷新页面后 `AuthGuard` 认为已登录就放行了，但菜单数据是空的。解决办法是把权限列表同步存储到 `localStorage`，同时在路由守卫里增加权限数据为空的拦截。"

**这段 1 分钟左右，是基本功展示，不过度展开。**

#### ① 地址转坐标的哈希算法

```ts
// utils/map.ts — 地址转坐标的哈希算法
function hashString(str: string): number {
  let hash = 0;
  for (let i = 0; i < str.length; i++) {
    const char = str.charCodeAt(i);
    hash = ((hash << 5) - hash) + char;
    hash = hash & hash;
  }
  return hash;
}

function addressToLngLat(city: string, address: string): [number, number] {
  const center = cityCenters[city];          // 城市中心坐标
  const hash = hashString(address);           // 对地址字符串做哈希
  const lngOffset = (Math.abs(hash) % 1600) / 10000 - 0.08;  // 产生 ±8km 的偏移
  const latOffset = (Math.abs(hash >> 8) % 1600) / 10000 - 0.08;
  return [center[0] + lngOffset, center[1] + latOffset];
}
```

**代码解释**：
- 订单列表只有文本地址（如"长沙市岳麓区麓谷大道"），没有真实经纬度
- 不可能为 Mock 数据调高德地理编码 API（要钱）
- 所以设计了这套哈希算法：以城市中心为基准，用地址字符串的 hash 值作为偏移量
- 同一个地址 hash 值固定，所以每次计算出来的坐标是一致的，模拟真实分布
- 没有这个算法，地图上就是一个点，看起来像假的；有了它，订单分布看起来是自然的

#### ② 动态 zoom 计算（根据距离自动调整）

```ts
// utils/map.ts — Haversine 距离公式
export function calculateDistance(p1: [number, number], p2: [number, number]): number {
  const R = 6371000; // 地球半径（米）
  const toRad = (deg: number) => (deg * Math.PI) / 180;
  const dLat = toRad(p2[1] - p1[1]);
  const dLng = toRad(p2[0] - p1[0]);
  const a =
    Math.sin(dLat / 2) * Math.sin(dLat / 2) +
    Math.cos(toRad(p1[1])) * Math.cos(toRad(p2[1])) *
    Math.sin(dLng / 2) * Math.sin(dLng / 2);
  const c = 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1 - a));
  return R * c;
}

// 根据距离返回合适的地图缩放级别
export function getZoomByDistance(distanceMeters: number): number {
  if (distanceMeters < 500) return 17;      // < 500m：街道级
  if (distanceMeters < 2000) return 16;     // < 2km：街区级
  if (distanceMeters < 5000) return 15;     // < 5km：市区级
  if (distanceMeters < 15000) return 14;    // < 15km：城市级
  if (distanceMeters < 50000) return 13;    // < 50km：区域级
  if (distanceMeters < 100000) return 12;   // < 100km：大区域
  if (distanceMeters < 300000) return 10;   // < 300km：省级
  return 9;                                 // > 300km：国家级
}
```

**代码解释**：
- 之前固定 `zoom: 13`，打开轨迹图看到的是整个省份，看不清市内道路
- 解决办法：先算起点和终点的直线距离（Haversine 公式），再根据距离动态返回 zoom 级别
- 距离越短，zoom 越大（越细致），确保市内短途能看到街道细节
- 这是地理信息系统（GIS）里的基础算法，答辩时可以说出来显得有深度

#### ③ 高德驾车路线规划 API（真实轨迹）

```ts
// utils/map.ts — 调用高德驾车路线规划 API
export async function fetchRealRoutePath(
  start: [number, number],
  end: [number, number],
): Promise<[number, number][]> {
  try {
    const url =
      `https://restapi.amap.com/v3/direction/driving?` +
      `key=${AMAP_KEY}` +
      `&origin=${start[0]},${start[1]}` +
      `&destination=${end[0]},${end[1]}` +
      `&extensions=all` +           // 获取完整路线坐标
      `&strategy=0`;                // 速度优先

    const res = await fetch(url);
    const data = await res.json();

    if (data.status === '1' && data.route?.paths?.length > 0) {
      const points: [number, number][] = [];
      const steps = data.route.paths[0].steps as any[];

      for (const step of steps) {
        const coords = step.polyline.split(';');
        for (const coord of coords) {
          const [lng, lat] = coord.split(',').map(Number);
          if (lng && lat) {
            points.push([lng, lat]);
          }
        }
      }

      // 如果点数太多，适当抽稀（每 3 个点取 1 个），避免渲染压力
      if (points.length > 200) {
        return points.filter((_, i) => i % 3 === 0);
      }
      return points;
    }
    throw new Error('API 返回空路线');
  } catch (err) {
    console.warn('高德路线 API 调用失败，降级为模拟路线:', err);
    return generateRoutePath(start, end, 12);
  }
}
```

**代码逐行解释**：

| 行 | 解释 |
|---|------|
| `key=${AMAP_KEY}` | 高德 Web 服务 API Key（注意：不是 JS API Key！这是踩过的坑） |
| `origin` / `destination` | 起点和终点经纬度，格式 `lng,lat` |
| `extensions=all` | 重要！不加这个只返回概要，加了才返回完整的 polyline 坐标点 |
| `strategy=0` | 路线策略：0 = 速度最快（还有 1=费用最低、2=距离最短等） |
| `data.route.paths[0].steps` | 高德返回的是"步骤"数组，每个步骤有一段 polyline |
| `step.polyline.split(';')` | 每段 polyline 是 `lng,lat;lng,lat;...` 格式，用 `;` 分割 |
| `points.filter((_, i) => i % 3 === 0)` | 抽稀算法：如果 API 返回几百个点，全画出来性能差，每 3 个点取 1 个 |
| `catch` 降级 | 如果 API 调用失败（Key 失效/超限制/网络错误），自动降级为 `generateRoutePath` 模拟路线，功能不中断 |

**技术深度——答辩时可以讲**：
1. **为什么不用直线？** —— 直线轨迹看起来像"穿楼"，不真实；真实道路是弯曲的
2. **为什么不用模拟算法？** —— 正弦波模拟的路线可能穿过河流/建筑，高德 API 返回的是真实可通行的道路
3. **API 限制** —— 高德个人开发者每天 5000 次调用，答辩演示完全够用；如果超限，代码里已经写了自动降级
4. **为什么用 `fetch` 而不是 `axios`？** —— 在工具函数里用原生 `fetch` 更轻量，不需要引入 axios；而且高德 API 是跨域的，`fetch` 直接能用（高德支持 JSONP 或 CORS）

## ⭐ 第 8 页 / 技术亮点①：Web Worker 异步导出 Excel

> **PPT 内容**：数据流图（Worker → postMessage → 生成 → 下载）

### 讲什么

这是最能体现**技术深度**的一个点——多线程编程。

### 关键代码

```ts
// export.worker.ts — 完整的 Web Worker 文件
import * as XLSX from 'xlsx';

addEventListener('message', (e) => {
  const { data, filename } = e.data;

  // 1. JSON 数据 → Excel Sheet
  const ws = XLSX.utils.json_to_sheet(data);

  // 2. 自动计算列宽
  const range = XLSX.utils.decode_range(ws['!ref']);
  for (let c = range.s.c; c <= range.e.c; c++) {
    let maxLen = 0;
    for (let r = range.s.r; r <= range.e.r; r++) {
      const cell = ws[XLSX.utils.encode_cell({ r, c })];
      if (cell?.v != null) maxLen = Math.max(maxLen, String(cell.v).length);
    }
    colWidths.push({ wch: Math.min(Math.max(maxLen + 4, 10), 60) });
  }

  // 3. 生成 ArrayBuffer 并传回主线程
  const arrayBuffer = XLSX.write(wb, { bookType: 'xlsx', type: 'array' });
  postMessage(
    { success: true, data: arrayBuffer, filename },
    { transfer: [arrayBuffer] }  // 零拷贝！
  );
});
```

**代码逐行解释**：

| 行 | 解释 |
|----|------|
| `import * as XLSX from 'xlsx'` | Worker 里也能 import npm 包！Vite 构建时自动打包 |
| `addEventListener('message', ...)` | Worker 的入口，主线程通过 `postMessage` 发数据过来 |
| `json_to_sheet(data)` | SheetJS 的核心 API，直接把 JSON 数组转成 Excel 表格 |
| `decode_range / encode_cell` | 遍历每个单元格计算内容长度，自动设置列宽 |
| `XLSX.write(wb, { type: 'array' })` | 输出为 ArrayBuffer——纯二进制数据，Worker 环境可用 |
| `{ transfer: [arrayBuffer] }` | **零拷贝传输**——ArrayBuffer 是 Transferable 对象，从 Worker 到主线程不复制内存 |


### 为什么用 Web Worker？

**核心原理**：
```
点击导出 → 创建 Worker → postMessage(数据)
                              ↓
                        Worker 线程接收
                        xlsx 库处理数据（不阻塞 UI）
                              ↓
                        postMessage(ArrayBuffer)
                              ↓
                        主线程接收 → Blob → 下载
```

**对比普通实现**：
```
点击导出 → 主线程处理大数据（页面卡死 2-3 秒）→ 下载
```

**技术深度点——答辩时可以讲**：
1. **为什么不直接在主线程做？** —— xlsx 库处理几万条订单时，主线程 JS 执行会阻塞渲染，页面卡死 2-3 秒，用户体验极差
2. **Web Worker 的原理**：浏览器在后台开辟一个独立线程，有自己独立的 V8 引擎实例，与主线程互不干扰
3. **Worker 的限制**：不能访问 DOM、不能操作 window/document、不能跨域，但可以 import npm 包
4. **Vite 的 Worker 支持**：`new URL('./worker.ts', import.meta.url)` 这个语法 Vite 会静态分析，构建时自动把 worker.ts 打包成一个独立 chunk
5. **零拷贝传输**：ArrayBuffer 是 Transferable，一个线程放弃所有权，另一个线程直接接管内存，完全不需要复制
6. **bundle 分离**：xlsx 库 276KB 只打包在 Worker chunk 里，不进入主 bundle，首屏加载体积减少

**口述稿**：
> "订单列表有个导出 Excel 功能。最初只是一个空壳——点一下弹个'导出成功'的消息。但真正实现导出的时候面临一个问题：xlsx 库处理几万条数据，如果在主线程执行，页面会卡死。
>
> 我的方案是用 Web Worker。点击导出时，创建 Worker 线程，把订单数据通过 postMessage 发给它。Worker 在后台独立线程里用 xlsx 库生成 Excel，生成完把 ArrayBuffer 传回来，主线程收到后创建 Blob 触发下载。整个过程页面完全不会卡顿。
>
> 而且这里有一个优化点：ArrayBuffer 是 Transferable 对象，postMessage 传它的时候可以用零拷贝机制——Worker 放弃这块内存的所有权，主线程直接接管，大文件传输不用复制。
>
> 还有一个额外的好处：xlsx 库被打包在 Worker chunk 里，276KB，不进入主 bundle，首屏加载体积反而小了。这就是为什么说 Web Worker 不仅是性能优化手段，也是代码分割的一种方式。"

**这段 1.5 分钟，是全场含金量最高的技术点，一定要把原理讲透。**

## ⭐ 第 9 页 / 技术亮点②：Framer Motion 页面动画

> **PPT 内容**：代码对比 + 效果描述 + **微交互示意图**

### 讲什么

从用户体验的维度提升项目质感。

### 关键代码

```tsx
// MainLayout.tsx — 页面过渡动画
<AnimatePresence mode="wait">      {/* 模式：等退出动画播完再播进入 */}
  <motion.div
    key={location.pathname}         {/* key 变化时触发进出场 */}
    initial={{ opacity: 0, y: 12 }} /* 初始状态：透明 + 下移 12px */
    animate={{ opacity: 1, y: 0 }}  /* 进入动画：不透明 + 归位 */
    exit={{ opacity: 0, y: -12 }}   /* 退出动画：透明 + 上移 12px */
    transition={{ duration: 0.15, ease: 'easeOut' }}
  >
    <Outlet />  {/* React Router 的渲染出口 */}
  </motion.div>
</AnimatePresence>
```

**代码逐行解释**：
- `AnimatePresence`：framer-motion 的核心组件，管理子元素的进出场动画，子元素要加 `key`
- `mode="wait"`：等退出动画播完（150ms），再播进入动画，避免重叠时的布局闪烁
- `key={location.pathname}`：路由切换时 pathname 变化，React 认为是不同的元素，触发动画
- `duration: 0.15`：150ms，比人眼感知阈值略长，既能感觉到顺滑又不拖沓
- `ease: 'easeOut'`：减速曲线——开始快结束慢，视觉上最自然

```tsx
// Dashboard — 统计卡片错峰入场
{statCards.map((card, index) => (
  <motion.div
    initial={{ opacity: 0, y: 24 }}
    animate={{ opacity: 1, y: 0 }}
    transition={{ delay: 0.08 * index, duration: 0.4 }}
    whileHover={{ y: -4 }}
  >
    <Card>...</Card>
  </motion.div>
))}
```

**代码解释**：
- `delay: 0.08 * index`：第 1 张卡 delay 0s，第 2 张 delay 0.08s，第 3 张 0.16s，第 4 张 0.24s
- 效果就是卡片从左到右依次弹出，不是一起冒出来——这就是"错峰入场"
- `whileHover={{ y: -4 }}`：鼠标悬停时卡片上浮 4px，配合卡片自己的 `hoverable` 属性，手感更好

```tsx
// Dashboard — 图标弹簧旋转动画
<motion.div
  whileHover={{ rotate: 10, scale: 1.1 }}
  transition={{ type: 'spring', stiffness: 300 }}
>
  {card.icon}
</motion.div>
```

**代码解释**：
- `type: 'spring'`：弹簧动画，不是匀速的 easing，而是有回弹效果
- `stiffness: 300`：弹簧刚度，数字越大回弹越快
- 鼠标悬停时图标旋转 10° 并放大 1.1 倍，松手后弹簧回正

### 技术深度——答辩可以说

1. **Framer Motion 相比于 CSS animation 的优势**：声明式、Spring 物理动画、AnimatePresence 进出场自动管理、JS 驱动的缓动曲线更可控
2. **duration 为什么选 150ms**：Google Material Design 研究指出，界面动画最佳时长 100-200ms，太短没感觉，太长嫌慢
3. **easeOut 曲线 vs easeInOut**：进入动画应该用 easeOut（快进慢出），退出动画用 easeIn（慢进快出），各有适用场景

**口述稿**：
> "Framer Motion 是一个生产级动画库，我在项目里用在了两个地方。
>
> 第一是页面切换。没用 Framer Motion 之前，切页面是生硬的闪现。加上 AnimatePresence 之后，内容会以 150ms 的淡入+上移效果平滑过渡，用户能感觉到页面在"流动"而不是"跳转"。
>
> 第二是 Dashboard 的卡片入场。第一次进入工作台时，四张统计卡片不是一起出现，而是从左到右依次弹出，每张间隔 80ms。这个叫错峰入场，能让用户的眼睛有时间聚焦到每一张卡片上。
>
> 此外还有微交互：比如图标 hover 时带弹簧旋转效果、卡片悬停时轻微上浮。这些细节单个看不算什么，但叠加起来给人的感觉就是"这个项目有质感"。"

## ⭐ 第 10 页 / 技术亮点③：实时 Dashboard + 数字跳动

> **PPT 内容**：计数器动效图 + 数据流 + **Haversine 公式说明**

### 讲什么

Dashboard 的数字"自己会动"，而且是根据真实距离动态缩放地图。

### 关键代码

```ts
// useCountUp.ts — 数字跳动 Hook（核心！）
export function useCountUp(target: number, duration = 600): number {
  const [value, setValue] = useState(target);
  const rafRef = useRef<number>();

  useEffect(() => {
    if (rafRef.current) cancelAnimationFrame(rafRef.current);
    if (target === value) return;

    const from = value;
    const startTime = performance.now();

    const animate = (now: number) => {
      const elapsed = now - startTime;
      const progress = Math.min(elapsed / duration, 1);
      const eased = 1 - Math.pow(1 - progress, 4);  // ease-out quartic
      const current = Math.floor(from + (target - from) * eased);

      if (progress < 1) {
        setValue(current);
        rafRef.current = requestAnimationFrame(animate);
      } else {
        setValue(target);
      }
    };

    rafRef.current = requestAnimationFrame(animate);
    return () => { if (rafRef.current) cancelAnimationFrame(rafRef.current); };
  }, [target, duration]);

  return value;
}
```

**逐行深度解释**：

| 行/概念 | 解释 |
|---------|------|
| `requestAnimationFrame(animate)` | 浏览器的帧回调函数，每秒调用 60 次（跟屏幕刷新率同步），比 `setInterval(fn, 16)` 更省资源——页面切到后台时会自动暂停 |
| `eased = 1 - (1 - progress)⁴` | **ease-out quartic（四次方缓出）**：一开始变化很快，越往后越慢，视觉上数字"冲上去再慢慢停"，让人清楚地看到变化过程 |
| `target === value` 提前 return | 如果目标值和当前值一样（比如数据没有变化），直接跳过，省去不必要的动画 |
| `cancelAnimationFrame` 清理 | 组件卸载时取消动画，防止内存泄漏；target 变化时取消上一次动画，重新开始 |

```typescript
// 在 Dashboard 中的使用
const animDriverCount = useCountUp(stats.driverCount); // 司机数跳动
const animRevenue = useCountUp(stats.totalRevenue);     // 流水跳动
const animOrders = useCountUp(stats.totalOrders);        // 订单数跳动
const animCities = useCountUp(stats.cityCount);          // 城市数跳动

// 每 3 秒自动微调数据（模拟真实业务波动）
useEffect(() => {
  const interval = setInterval(() => {
    setStats(prev => ({
      driverCount: prev.driverCount + Math.floor((Math.random() - 0.45) * 30),
      totalRevenue: prev.totalRevenue + Math.floor((Math.random() - 0.45) * 60000),
      totalOrders: prev.totalOrders + Math.floor((Math.random() - 0.45) * 800),
      cityCount: prev.cityCount + (Math.random() > 0.92 ? 1 : 0),
    }));
    
    setCharts(prev => ({
      ...prev,
      orderTrend: {
        months: prev.orderTrend.months,
        orders: prev.orderTrend.orders.map(v =>
          Math.max(10000, v + Math.floor((Math.random() - 0.4) * 400))
        ),
        revenue: prev.orderTrend.revenue.map(v =>
          Math.max(300, v + Math.floor((Math.random() - 0.4) * 40))
        ),
      },
      // ... 其他图表数据也自动微调
    }));
  }, 3000);
  
  return () => clearInterval(interval);
}, []);
```

### 技术深度

1. **为什么不直接用 setInterval 更新数字？** —— 因为 setInterval 不能控制变化过程，数字会跳变而不是平滑过渡
2. **requestAnimationFrame 的优势**：由浏览器的渲染循环驱动，页面不可见时自动暂停，不耗性能
3. **ease-out quartic 为什么选 4 次方**：线性变化太生硬，二次方变化不够明显，四次方在视觉上最舒适——数字快速冲到接近目标，然后慢慢微调到准确值
4. **性能考虑**：`setValue` 每帧调用会触发 React 重渲染，但因为变化的是数字文本而不是 DOM 结构，React 的 diff 算法只需更新文本节点，性能开销极小

**口述稿**：
> "工作台的四个核心数字，我做了一个自动跳动效果。数据每 3 秒自动微调一波，模拟真实业务波动——司机数量上下浮动、流水和订单量变化，图表数据也跟着动态更新。
>
> 但直接更新数字会跳变，看起来非常生硬。所以我实现了一个 useCountUp 自定义 Hook，核心原理是 requestAnimationFrame 驱动的缓动动画。
>
> 具体来说：旧值是 1248，新值是 1256，相差 8。Hook 会在 600 毫秒内，用 ease-out quartic 曲线把数字从 1248 平滑过渡到 1256。一开始变化快，后面慢慢微调到位，视觉上非常自然。
>
> 这个 Hook 还实现了目标值频繁变化时的正确处理——如果旧动画没播完新值又来了，旧的取消、从当前位置开始新的动画。这样数据每 3 秒更新一次，数字也跟着平滑跳动，不会卡顿也不会跳变。"

**这段 1 分钟，核心讲清楚 requestAnimationFrame 和缓动曲线。**

## 第 12 页 / 踩坑与解决

> **PPT 内容**：3 个踩坑点的时间线

### ① React 版本兼容性问题

**问题**：Vite 默认安装 React 19，但 AntD 5.2 内部依赖 `findDOMNode` 这个旧 API，React 19 已移除，导致所有 Tooltip、Modal、Dropdown 崩溃。

**解决**：将 React + React-DOM 锁定到 18.3.1。

```json
"react": "^18.3.1",
"react-dom": "^18.3.1"
```

**口述稿**：
> "第一个坑是 Vite 初始化时默认装了 React 19，但 AntD 5.2 还在用 findDOMNode，这个 API 在 React 19 里被移除了。结果所有弹窗和下拉菜单全部崩溃，控制台全是报错。排查后把 React 降级到 18.3.1，问题解决。"

### ② 高德地图 Key 选型错误

**问题**：开始申请高德 Key 时选了"Web 服务"，但那个是用来给后端调逆地理编码接口的，前端浏览器渲染地图应该选"Web端 JS API"。选错 Key 地图直接白屏。

**解决**：重新申请 JS API 的 Key。后来又申请了 Web 服务 Key 用于驾车路线规划 API。

**口述稿**：
> "第二个坑是高德地图的 Key。高德的 Key 分两种：Web 服务是给后端用的，Web端(JS API)才是给前端浏览器用的。第一次我申请错了服务类型，地图白屏一整天。
>
> 后来接驾车路线规划 API 的时候，又发现需要 Web 服务 Key，所以项目里现在有两个 Key：JS API Key 用于前端渲染地图，Web 服务 Key 用于后端接口调用。"

### ③ Token 持久化 + 权限状态不一致

**问题**：刷新页面后 `token` 从 localStorage 恢复，但 `permissionList` 丢失 → `AuthGuard` 放行（因为 token 存在）→ 页面进了但菜单渲染不出来。

**解决**：把 `permissionList` 也持久化，`AuthGuard` 增加 permission 为空的拦截。

### ④ toFixed 报错

**问题**：添加订单后，价格字段可能是字符串或 undefined，直接调用 `.toFixed(2)` 会报 `TypeError: p.toFixed is not a function`。

**解决**：所有价格显示处改为 `Number(p || 0).toFixed(2)`，防止类型错误。

**口述稿**：
> "第三个坑是权限状态和 token 不一致。刷新页面后 token 从 localStorage 恢复，但权限列表是空的，AuthGuard 以为已登录就放行了，进去之后菜单是空白的。我的解决是双管齐下：一是把权限列表也持久化到 localStorage，二是在路由守卫里增加权限数据为空的判断，即使 token 存在，权限没加载也不给进。
>
> 还有一个小的坑：添加订单时价格字段有时候是字符串，直接调 toFixed 会报错。后来全部改成 Number(p || 0).toFixed(2) 的防御性写法，彻底解决。"

**这段 40 秒，每个坑 10-15 秒，不要展开太多细节。**

## 🎯 附录：常见追问应对

### Q：为什么不用 Redux / Zustand 而用 Context？

> "项目规模中等，Context + Reducer 已经能满足需求。Redux 的样板代码在这个体量下是过度设计。但如果项目继续扩大、需要跨组件频繁通信，我会考虑切换到 Zustand——它比 Redux 轻量，API 简洁，而且不需要 Provider 包裹。"

### Q：Mock 数据怎么切换到真实后端？

> "项目已经做好了切换准备。API 层分为 `api/index.ts`（接口定义）和 `api/mock.ts`（Mock 数据）。目前页面里直接 import 了 mock 数据。真实场景下，只需要替换为 axios 调用真实接口，`request.ts` 的拦截器已经配好了 token 注入和 401 处理。"

### Q：你这个项目做了多久？

> "从项目初始化到现在的功能，陆续花了大概两周时间。主要时间花在 RBAC 权限设计、地图集成和高德 API 接入上。"

### Q：Web Worker 有哪些限制？

> "Worker 不能访问 DOM，不能操作 window/document，不能跨域请求（可以用 importScripts 或者通过主线程代理）。与主线程通过 postMessage 通信，数据是拷贝（对于普通对象）或转移（对于 ArrayBuffer）的。"

### Q：Framer Motion 和 CSS 动画比有什么优势？

> "CSS 动画适合简单的 hover 效果，但涉及到复杂的编排——比如错峰入场、页面进出场、弹簧物理动画——Framer Motion 的声明式 API 写起来更直观。而且 AnimatePresence 能自动管理元素的挂载和卸载过程，CSS 实现同样的效果需要配合 React 的生命周期，代码会复杂很多。"

### Q：高德路线规划 API 的 limit 怎么办？

> "高德个人开发者每天 5000 次调用，答辩演示完全够用。而且我在代码里写了自动降级逻辑——如果 API 调用失败（Key 失效/超限制/网络错误），自动降级为 generateRoutePath 模拟路线，功能不中断。如果以后商用，可以申请企业开发者，每天 300 万次调用。"

### Q：你在项目中遇到的最大困难是什么？

> "地图集成遇到的坑最多。主要是高德文档比较老，网上搜到的解决方案和新版 API 对不上。比如 MarkerCluster 插件加载方式变了，按旧文档写的代码跑不起来，需要自己试错。还有就是驾车路线规划 API 需要单独的 Web 服务 Key，和前端渲染用的 JS API Key 不一样，这个坑了我大半天。"

### Q：这个项目还有什么可以改进的地方？

> "如果时间充裕，我还会加三个功能：一是单元测试，核心 Hook 和工具函数没有覆盖测试；二是 Error Boundary，现在任何一个组件 crash 都会导致整个后台白屏，加了之后能优雅降级；三是虚拟滚动，订单列表如果真有上万条数据，现在的渲染方式会卡，可以用 react-window 做虚拟滚动。"
